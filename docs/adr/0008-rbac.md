# ADR-0008 — RBAC via `has_permission` security-definer

- **Status:** Aceito
- **Data:** 2026-06-21
- **Escopo:** Plataforma

## Contexto
Storage de roles em colunas de profile causa recursão de RLS e escalada de privilégio. Permissões hardcoded em código quebram extensibilidade. Multi-store exige escopo de permissão por loja.

## Decisão
Modelar autorização em quatro tabelas + funções security-definer:
- `roles(code, name, ...)` — papéis de plataforma (super_admin, store_admin, ops_manager, fulfillment_operator, support_agent, finance, customer).
- `permissions(code, ...)` — verbos finos por recurso (`orders.write`, `orders.refund`, etc.).
- `role_permissions(role_id, permission_id)` — quais permissões cada papel tem.
- `user_roles(user_id, role_id, store_id NULL)` — atribuição com escopo de loja (NULL = global).

Funções de consulta (todas SECURITY DEFINER + `search_path=public` fixo):
- `has_role(uid, code, store NULL)`
- `has_permission(uid, code, store NULL)`
- `is_super_admin(uid)`
- `user_store_ids(uid)`
- `current_user_context()`

RLS policies usam **somente** essas funções, nunca SELECT direto em `user_roles`/`role_permissions` (evita recursão).

## Alternativas consideradas
1. Coluna `role` em `profiles` → rejeitado: privilege escalation + recursão.
2. JWT claims com roles → rejeitado: stale claims, sem revogação imediata.
3. Casbin / OPA externo → rejeitado: latência e complexidade desnecessária.

## Consequências
- ✅ Permissões granulares e escopadas por loja.
- ✅ Sem recursão de RLS.
- ✅ Novo papel/permissão é insert, não deploy.
- ⚠️ Funções SECURITY DEFINER exigem revisão de `search_path` em cada criação.
- ⚠️ Promoção do primeiro super_admin via `claim_first_super_admin` (one-shot).

## Invariantes
- Nenhuma tabela armazena role como string em coluna de usuário/profile.
- Toda RLS policy que precisa de role usa `has_permission` ou `has_role`.
- Toda função security-definer declara `SET search_path = public`.
