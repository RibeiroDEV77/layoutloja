# ADR-0009 — RLS por permissão + `store_id` (multi-tenant)

- **Status:** Aceito
- **Data:** 2026-06-21
- **Escopo:** Plataforma

## Contexto
Multi-tenant em Postgres pode ser implementado por schema-por-tenant, banco-por-tenant, ou row-level com `tenant_id`. As duas primeiras opções não escalam operacionalmente (milhares de tenants); a terceira sem RLS bem feita vaza dados.

## Decisão
Adotar **row-level multi-tenancy**:
- Toda tabela de negócio tem `store_id uuid NOT NULL`.
- RLS habilitada com policies que combinam:
  - `is_super_admin(auth.uid())` → bypass cross-store
  - `has_permission(auth.uid(), '<code>', store_id)` → operador da loja
  - `<resource>_owner(auth.uid(), id)` → cliente final (quando aplicável)
- GRANTs explícitos a `authenticated` e `service_role` em **toda** tabela criada no schema `public` (PostgREST não concede defaults).
- Workers e webhooks usam `service_role` que bypassa RLS — exigem auditoria própria.

Padrão por tabela:
```sql
CREATE TABLE public.<t>(...);
GRANT SELECT, INSERT, UPDATE, DELETE ON public.<t> TO authenticated;
GRANT ALL ON public.<t> TO service_role;
-- (opcional) GRANT SELECT ON public.<t> TO anon WHEN política público-leitura
ALTER TABLE public.<t> ENABLE ROW LEVEL SECURITY;
CREATE POLICY ... USING (...) WITH CHECK (...);
```

Helpers comuns: `<resource>_store_id(_id)`, `is_<resource>_owner(_id)`, `customer_store_id`, `cart_store_id`, `order_store_id`, etc.

## Alternativas consideradas
1. Schema-per-tenant → rejeitado: catalog bloat, migrations N×.
2. Tenancy via app-layer apenas (filtro no service) → rejeitado: erro humano vaza tudo.
3. PostgreSQL Row Security sem helpers (subqueries diretas) → rejeitado: recursão e performance ruim.

## Consequências
- ✅ Isolamento mesmo se um service esquecer filtro `where store_id = ...`.
- ✅ Super admin atende qualquer loja sem mudar contexto de conexão.
- ✅ Suporte natural a usuário com acesso a múltiplas lojas.
- ⚠️ Queries cross-store exigem `is_super_admin` ou agregação em service privilegiado.
- ⚠️ Esquecer GRANT após CREATE TABLE quebra Data API silenciosamente — coberto por checklist de migration.

## Invariantes
- Toda tabela em `public.*` de negócio tem `store_id NOT NULL` (exceção: tabelas globais de plataforma — roles, permissions, system_settings).
- Toda CREATE TABLE migration vem acompanhada de GRANT + ENABLE RLS + ao menos uma POLICY na mesma migration.
- Nenhuma policy faz SELECT na própria tabela (use security-definer helper).
- Nenhum service usa `service_role` para leitura comum (apenas operações privilegiadas auditadas).
